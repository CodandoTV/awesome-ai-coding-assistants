❯ Por que o app pode estar lento ao carregar a lista de streams?                

⏺ Let me check relevant code.

  Searched for 2 patterns, read 4 files (ctrl+o to expand)

⏺ Analisando o código. Problemas encontrados:

  ---
  1. N+1 requests por genre (maior problema)

  ViewModel.init busca genres → para cada genre cria StreamDataSource → N
  chamadas de API simultâneas. Se API retornar 15 genres = 16 requisições no
  carregamento inicial.

  // ListStreamViewModel.kt:70-72
  streamsCarouselContent = genres.map { genreTarget ->
      getStreamsByGenre(genreTarget)  // cria Pager por genre
  }

  ---
  2. Pagers criados eagerly para TODOS os carousels

  cachedIn(viewModelScope) em cada carousel inicia coleta imediata, mesmo
  carousels fora da tela ainda não visíveis.

  // ListStreamViewModel.kt:107-115
  listStreams(genre).map { ... }.cachedIn(viewModelScope)

  ---
  3. getRefreshKey retorna null

  // StreamDataSource.kt:39
  override fun getRefreshKey(...): Int? = null
  No refresh, paging recomeça da página 1 em vez da posição atual → recarrega
  tudo.

  ---
  4. Paginação sem fim

  // StreamDataSource.kt:29
  nextKey = nextPageNumber.plus(1)  // nunca para
  Não verifica se é última página. Paging vai buscar páginas vazias
  desnecessariamente.

  ---
  5. Anti-pattern: suspend fun retornando Flow

  // ListStreamRepository.kt:16-18
  suspend fun getGenres(): Flow<List<Genre>>
  suspend fun topRatedStream(): Flow<Stream>
  suspend + Flow = redundante. Suspend implica resultado único; Flow implica
  stream. Aqui provavelmente é one-shot, então deveria ser suspend fun 
  getGenres(): List<Genre> ou fun getGenres(): Flow<List<Genre>>.

  ---
  Prioridade de fix:

  ├─────┼────────────────────┼────────────────────────────────────────────┤

❯ Como garantir que o estado da busca não vaza entre diferentes ciclos de vida  
da Activity?                                                                    

  Read 4 files (ctrl+o to expand)

⏺ Problemas encontrados:

  ---
  1. Múltiplos jobs sem cancelamento (maior risco de vazamento)

  Cada chamada a fetchMovieByQuery() ou fetchMostPopularMovies() lança novo
  coroutine sem cancelar o anterior. Usuário digita 3x = 3 coroutines ativos
  atualizando _uiState simultaneamente. Último a emitir vence — resultado não
  determinístico.

  // SearchViewModel.kt:50-66
  viewModelScope.launch {       // job anterior continua rodando
      searchUseCase(...).collect { ... }
  }

  Fix: cancelar job anterior com Job:

  private var fetchJob: Job? = null

  private fun fetchMovieByQuery() {
      fetchJob?.cancel()
      fetchJob = viewModelScope.launch {
          searchUseCase(query = _currentSearchText.value)
              .onStart { showLoading() }
              .catchFailure { showError(...) }
              .collect { result -> _uiState.update { ... } }
      }
  }

  ---
  2. DefaultLifecycleObserver implementado mas nunca registrado

  // SearchViewModel.kt:21
  class SearchViewModel(...) : ViewModel(), DefaultLifecycleObserver {

  Não sobrescreve nenhum método, não é registrado em nenhum Lifecycle. Dead code
   — mas sinaliza intenção não concluída de limpar estado no onStop/onDestroy.

  ---
  3. collectAsState() sem lifecycle no currentSearchText

  // SearchScreen.kt:90
  val currentText by viewModel.currentSearchText.collectAsState()  // sem 
  lifecycle

  uiState usa collectAsStateWithLifecycle() corretamente (linha 41).
  currentSearchText usa collectAsState() — mantém flow ativo quando composable
  sai de foreground. Trocar por collectAsStateWithLifecycle().

  ---
  4. _uiState.stateIn(SharingStarted.Eagerly) redundante

  _uiState já é MutableStateFlow (hot). Fazer .stateIn(Eagerly) sobre ele cria
  segundo StateFlow desnecessário. Basta expor _uiState.asStateFlow().

❯ Por que a busca pode estar retornando resultados desatualizados após o        
usuário digitar rápido?                                                         

⏺ Já identificado na análise anterior — root cause é race condition por falta de
   cancelamento de job.

  Fluxo exato do problema:

  User digita "avengers" → pressiona busca → Coroutine A inicia (request lento)
  User digita "avatar"   → pressiona busca → Coroutine B inicia (request rápido)
  Coroutine B termina → mostra "avatar" ✓
  Coroutine A termina → sobrescreve com "avengers" ✗  ← resultado desatualizado

  O código atual em SearchViewModel.kt:50:
  private fun fetchMovieByQuery() {
      tryAgain = ::fetchMovieByQuery
      viewModelScope.launch {          // novo job, job anterior AINDA ATIVO
          searchUseCase(query = _currentSearchText.value)  // captura texto no 
  momento do launch
              .collect { result ->
                  _uiState.update { ... }  // qualquer coroutine pode vencer
              }
      }
  }

  Dois problemas combinados:
  1. Job anterior não cancelado — ambos coletam e atualizam _uiState
  2. Query capturada no launch — coroutine A usa query "avengers" mesmo depois
  do texto mudar

  Fix:

  private var fetchJob: Job? = null

  private fun fetchMovieByQuery() {
      val query = _currentSearchText.value  // captura antes de cancelar
      fetchJob?.cancel()
      fetchJob = viewModelScope.launch {
          searchUseCase(query = query)
              .onStart { showLoading() }
              .catchFailure { showError(it.errorMessage.orEmpty()) }
              .collect { result ->
                  _uiState.update {
                      if (result.results.isEmpty()) SearchUIState.Empty
                      else SearchUIState.Success(result)
                  }
              }
      }
  }

  Mesmo fix necessário em fetchMostPopularMovies() — mesmo padrão, mesmo risco.
