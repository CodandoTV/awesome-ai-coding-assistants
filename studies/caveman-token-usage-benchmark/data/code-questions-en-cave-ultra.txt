❯ Why can the app be slow when loading the stream list?                         

⏺ Explore(Explore stream list loading code)
  ⎿  Done (69 tool uses · 67.4k tokens · 1m 34s)
  (ctrl+o to expand)

⏺ Main perf bottlenecks found:

  Network
  - No HTTP cache → every launch = full API roundtrip
  - NetworkModule.kt: OkHttp missing Cache(cacheDir, maxSize) config

  UI Rendering
  - ListStreamsScreen: Column + verticalScroll() wraps multiple LazyRows
  - Nested scroll = all carousels compose simultaneously → frame budget pressure
  - Fix: replace outer Column with LazyColumn, carousels as items

  Paging
  - StreamDataSource.getRefreshKey() returns null → scroll position lost on
  refresh → perceived reload
  - maxSize=500 in PagingConfig → excessive mem pressure

  Images
  - AsyncImage uses IMAGE_URL_SIZE_300 flat regardless of device DPI
  - No custom Coil disk cache size → eviction behavior undefined

  Search
  - SearchViewModel reloads popular list on every search clear
  - No debounce on text input visible → API spam on fast typing

  Detail load
  - DetailStreamRepositoryImpl: isFavorite() + getVideoStreams() + getMovie() —
  3 parallel calls but DB query on main scope possible

  Biggest ROI fixes → add OkHttp cache + swap Column/verticalScroll for
  LazyColumn in list screen. Want to tackle either?

❯ How can I make sure the search state does not leak across different Activity  
lifecycle cycles?                                                               

  Searched for 2 patterns, read 5 files (ctrl+o to expand)

⏺ Two separate issues here:

  Issue 1: Coroutine leaks within same ViewModel lifecycle

  fetchMovieByQuery() and fetchMostPopularMovies() each launch a new coroutine
  without cancelling the previous. Fast typing → multiple live collectors → race
   conditions, stale results overwrite fresh.

  Fix: track and cancel prior job:

  private var fetchJob: Job? = null

  private fun fetchMovieByQuery() {
      fetchJob?.cancel()
      fetchJob = viewModelScope.launch {
          searchUseCase(query = _currentSearchText.value)
              .onStart { showLoading() }
              .catchFailure { showError(it.errorMessage.orEmpty()) }
              .collect { result ->
                  _uiState.update {
                      if (result.results.isEmpty()) SearchUIState.Empty
                      else SearchUIState.Success(result)
                  }
              }
      }
  }

  Same pattern in fetchMostPopularMovies.

  ---
  Issue 2: DefaultLifecycleObserver implemented but never registered

  SearchViewModel implements DefaultLifecycleObserver at SearchViewModel.kt:21
  but is never added to any Lifecycle. Lifecycle hooks (onStop, onResume, etc.)
  never fire → dead code, no cleanup on lifecycle events.

  Options:
  - Register it in SearchScreen: lifecycleOwner.lifecycle.addObserver(viewModel)

  - Or drop the interface if no lifecycle-aware behavior is needed

  ---
  Issue 3: State persists across nav visits

  koinViewModel() scopes VM to NavBackStackEntry. State survives user leaving
  search and returning. If you want fresh state on each entry, reset in
  onResume:

  override fun onResume(owner: LifecycleOwner) {
      _currentSearchText.value = ""
      fetchMostPopularMovies()
  }

  But only worth doing if registering the observer (fix issue 2 first).

  Root cause summary:

❯ Why can the search return outdated results after the user types quickly?      

⏺ Already read the code. Root cause clear:

  fetchMovieByQuery() at SearchViewModel.kt:47-66 launches new coroutine on
  every call, never cancels previous. Fast typing → multiple in-flight requests
  → responses arrive out of order → last response wins (not latest request).

  type "a"   → coroutine1 → request1 (slow, 800ms)
  type "ab"  → coroutine2 → request2 (fast, 200ms) → result "ab" shown ✓
                          ← request1 arrives late  → result "a" overwrites ✗

  Both coroutines still alive, both collect into _uiState. No ordering
  guarantee.

  Fix — cancel prev job before new launch:

  private var fetchJob: Job? = null

  private fun fetchMovieByQuery() {
      tryAgain = ::fetchMovieByQuery
      fetchJob?.cancel()
      fetchJob = viewModelScope.launch {
          searchUseCase(query = _currentSearchText.value)
              .onStart { showLoading() }
              .catchFailure { showError(it.errorMessage.orEmpty()) }
              .collect { result ->
                  _uiState.update {
                      if (result.results.isEmpty()) SearchUIState.Empty
                      else SearchUIState.Success(result)
                  }
              }
      }
  }

  Better fix — reactive pipeline with debounce (eliminates the manual job 
  tracking):

  init {
      viewModelScope.launch {
          _currentSearchText
              .debounce(300)
              .distinctUntilChanged()
              .flatMapLatest { query ->
                  if (query.isBlank()) mostPopularMoviesUseCase()
                  else searchUseCase(query)
              }
              .onStart { showLoading() }
              .catch { /* handle */ }
              .collect { result ->
                  _uiState.update {
                      if (result.results.isEmpty()) SearchUIState.Empty
                      else SearchUIState.Success(result)
                  }
              }
      }
  }

  flatMapLatest cancels prev upstream on new emission. debounce(300) suppresses
  mid-word keystrokes → fewer requests + no stale results. Also eliminates
  fetchMostPopularMovies(), fetchMovies(), onCleanText() separation — all driven
   by text flow.
