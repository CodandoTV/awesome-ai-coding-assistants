❯ Por que o app pode estar lento ao carregar a lista de streams?                

⏺ Explore(Explore stream list loading code)
  ⎿  Done (56 tool uses · 56.5k tokens · 1m 30s)
  (ctrl+o to expand)

⏺ Aqui estão os principais motivos pelos quais o app pode estar lento ao
  carregar a lista de streams:

  ---
  Problemas de Performance Identificados

  1. verticalScroll em vez de LazyColumn (Alto Impacto)

  ListStreamsScreen.kt

  A tela usa um Column com verticalScroll para renderizar todos os carrosséis.
  Isso significa que todos os carrosséis são compostos ao mesmo tempo, mesmo os
  que estão fora da tela. Um LazyColumn resolveria isso com reciclagem de views.

  ---
  2. Múltiplas requisições paralelas por gênero (Alto Impacto)

  ListStreamViewModel.kt:104-117

  Cada gênero cria um PagingFlow independente com .cachedIn(viewModelScope). Se
  houver 10 gêneros, 10 requisições de rede disparam simultaneamente no startup
  — congestionamento de rede e lag na UI.

  ---
  3. getRefreshKey() retorna null (Alto Impacto)

  StreamDataSource.kt:39

  override fun getRefreshKey(...): Int? = null

  Isso faz o paging recomeçar do início em rotação de tela ou pull-to-refresh,
  invalidando o cache e gerando novas requisições desnecessárias.

  ---
  4. Search sem paginação (Alto Impacto)

  SearchScreen.kt:114-135

  Os resultados de busca são renderizados em um Column com .map() direto — sem
  LazyColumn, sem paginação. Com muitos resultados, o consumo de memória é
  ilimitado.

  ---
  5. maxSize = 500 no PagingConfig (Médio Impacto)

  ListStreamRepository.kt

  500 itens por carrossel × múltiplos gêneros = muita pressão de memória em
  dispositivos com pouca RAM.

  ---
  Nota Positiva

  A modificação em ListSearchStreamResponse.kt (visível no git status) que
  adiciona @Immutable é uma melhoria — evita recomposições desnecessárias no
  Compose.

  ---
  O maior ganho imediato seria substituir o Column + verticalScroll por
  LazyColumn na ListStreamsScreen e adiar o carregamento dos carrosséis fora da
  viewport.

❯ Como garantir que o estado da busca não vaza entre diferentes ciclos de vida  
da Activity?                                                                    

  Searched for 1 pattern, read 3 files (ctrl+o to expand)

⏺ Encontrei três problemas concretos no código. Vou descrever cada um com o
  trecho relevante:

  ---
  1. Coroutines concorrentes sem cancelamento (Race condition)

  SearchViewModel.kt:50 e :72

  Cada chamada a fetchMovieByQuery() ou fetchMostPopularMovies() lança um novo
  viewModelScope.launch sem cancelar o anterior. Se o usuário digitar rápido,
  várias coroutines ficam vivas simultaneamente atualizando _uiState em ordem
  imprevisível.

  Solução: usar um Job cancelável ou migrar para flatMapLatest:

  private var searchJob: Job? = null

  private fun fetchMovieByQuery() {
      searchJob?.cancel()
      searchJob = viewModelScope.launch {
          searchUseCase(query = _currentSearchText.value)
              .onStart { showLoading() }
              .catchFailure { showError(it.errorMessage.orEmpty()) }
              .collect { result -> _uiState.update { ... } }
      }
  }

  ---
  2. collectAsState() não respeita lifecycle

  SearchScreen.kt:90

  val currentText by viewModel.currentSearchText.collectAsState() // ← errado

  Enquanto uiState usa collectAsStateWithLifecycle() (linha 41), o
  currentSearchText usa collectAsState() — que continua coletando mesmo com o
  app em background. Deve ser consistente:

  val currentText by viewModel.currentSearchText.collectAsStateWithLifecycle()

  ---
  3. SharingStarted.Eagerly mantém o flow ativo para sempre

  SearchViewModel.kt:26-30

  val uiState = _uiState.stateIn(
      scope = viewModelScope,
      started = SharingStarted.Eagerly, // ← nunca pausa
      initialValue = _uiState.value
  )

  Eagerly inicia o compartilhamento imediatamente e nunca suspende, mesmo quando
   nenhuma tela está coletando. Para busca, o ideal é WhileSubscribed com um
  timeout curto para sobreviver a rotações de tela:

  started = SharingStarted.WhileSubscribed(5_000)

  ---
  4. DefaultLifecycleObserver implementado mas nunca registrado

  SearchViewModel.kt:21

  class SearchViewModel(...) : ViewModel(), DefaultLifecycleObserver

  A interface está implementada mas nenhum método (onStop, onDestroy, etc.) é
  sobrescrito, e o ViewModel nunca é registrado via
  lifecycle.addObserver(viewModel). É código morto que gera falsa sensação de
  que o ciclo de vida está sendo tratado.

❯ Por que a busca pode estar retornando resultados desatualizados após o        
usuário digitar rápido?                                                         

⏺ Isso é exatamente o problema do item 1 que identifiquei antes. O código em
  SearchViewModel.kt deixa o mecanismo claro:

  // linha 47-66
  private fun fetchMovieByQuery() {
      tryAgain = ::fetchMovieByQuery

      viewModelScope.launch {          // ← novo job lançado
          searchUseCase(query = _currentSearchText.value)  // ← captura o texto 
  AGORA
              .onStart { showLoading() }
              .collect { result ->
                  _uiState.update { ... }  // ← atualiza quando chegar
              }
      }
  }

  O problema: fetchMovies() (linha 39) é chamado a cada busca, lançando um novo
  launch sem cancelar o anterior. Com digitação rápida:

  usuário digita "ba"  → Job A lançado, query = "ba"
  usuário digita "bat" → Job B lançado, query = "bat"  (Job A ainda vivo)
  usuário digita "batm" → Job C lançado, query = "batm" (Jobs A e B ainda vivos)

  Chegada das respostas (rede é imprevisível):
    Job C responde primeiro  → UI mostra "batm" ✓
    Job A responde por último → UI mostra "ba"  ✗  ← resultado desatualizado

  ---
  Correção com flatMapLatest

  A solução idiomática em Kotlin é combinar o StateFlow do texto com
  flatMapLatest — que cancela automaticamente a busca anterior quando o texto
  muda:

  // Em vez de lançar jobs manualmente, reativa o flow quando o texto muda
  val uiState: StateFlow<SearchUIState> = _currentSearchText
      .debounce(300L)                         // evita disparar a cada tecla
      .filter { it.isNotBlank() }
      .flatMapLatest { query ->               // cancela a busca anterior
          searchUseCase(query = query)
              .map { result ->
                  if (result.results.isEmpty()) SearchUIState.Empty
                  else SearchUIState.Success(result)
              }
              .onStart { emit(SearchUIState.Loading) }
              .catch { emit(SearchUIState.Error(it.message.orEmpty())) }
      }
      .stateIn(
          scope = viewModelScope,
          started = SharingStarted.WhileSubscribed(5_000),
          initialValue = SearchUIState.Loading
      )

  Com isso:
  - debounce(300) evita requisição a cada tecla
  - flatMapLatest garante que só a busca mais recente fica ativa
  - Elimina a necessidade de gerenciar Job manualmente e o campo tryAgain

  O fetchMovies(), fetchMovieByQuery(), e fetchMostPopularMovies() seriam
  substituídos por esse pipeline declarativo.
