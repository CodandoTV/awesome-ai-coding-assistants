  6. maxSize=500 in Pager config — memory accumulates across genre carousels

  per-item on every load
  9. No withContext(Dispatchers.IO) — relies on Retrofit default, not explicit

  Biggest bang for fix: swap Column.verticalScroll → LazyColumn + add OkHttp
  cache. Those two alone should cut visible slowness significantly.

❯ How can I make sure the search state does not leak across different Activity  
lifecycle cycles?                                                               

  Searched for 2 patterns, read 2 files (ctrl+o to expand)

⏺ Two problems:

  1. DefaultLifecycleObserver implemented but never registered

  SearchViewModel implements DefaultLifecycleObserver (line 21) but none of the
  lifecycle methods (onStop, onDestroy, etc.) are overridden, and it's likely
  never registered via lifecycle.addObserver(viewModel) in Fragment/Activity.
  Interface is dead weight — no lifecycle callbacks fire.

  2. SharingStarted.Eagerly keeps flow hot forever

  uiState (line 26-30) uses Eagerly — starts collecting immediately, never
  stops, even when screen is off-stack.

  ---
  Fixes:

  A. Reset state on onCleared — ViewModel IS destroyed on real Activity finish
  (Koin viewModel {} scopes correctly). But add explicit cleanup:

  override fun onCleared() {
      super.onCleared()
      _currentSearchText.value = ""
      _uiState.value = SearchUIState.Loading
  }

  B. Switch to WhileSubscribed — stops upstream when no collectors (screen not
  visible):

  val uiState: StateFlow<SearchUIState> = _uiState.stateIn(
      scope = viewModelScope,
      started = SharingStarted.WhileSubscribed(5000),
      initialValue = SearchUIState.Loading
  )

  C. Either use DefaultLifecycleObserver properly or remove it. If goal is reset
   search text on back navigation:

  // In Fragment/Activity:
  lifecycle.addObserver(viewModel)

  // In ViewModel:
  override fun onStop(owner: LifecycleOwner) {
      onCleanText()
  }

  Or remove the interface entirely — it's misleading as-is.

  ---
  Root cause summary: Koin viewModel {} already scopes correctly to Activity —
  ViewModel IS cleared on Activity destroy. No true cross-lifecycle leak today.
  But Eagerly + no onCleared reset = stale state visible briefly on re-entry,
  and dead DefaultLifecycleObserver = false safety signal.

❯ Why can the search return outdated results after the user types quickly?      

⏺ Already have code from prior read. Root cause is clear:

  Race condition — no job cancellation.

  // SearchViewModel.kt:50
  viewModelScope.launch {          // ← new coroutine every call
      searchUseCase(query = _currentSearchText.value)
          ...
          .collect { result ->
              _uiState.update { ... }  // ← last-to-finish wins, not last-typed
          }
  }

  User types "ma", "mar", "mars" fast:
  1. Coroutine A → query "ma" (slow network)
  2. Coroutine B → query "mar"
  3. Coroutine C → query "mars"
  4. C finishes → shows "mars" results ✓
  5. A finishes last → overwrites with "ma" results ✗

  Fix — track and cancel previous job:

  private var searchJob: Job? = null

  private fun fetchMovieByQuery() {
      tryAgain = ::fetchMovieByQuery
      searchJob?.cancel()  // cancel in-flight request
      searchJob = viewModelScope.launch {
          searchUseCase(query = _currentSearchText.value)
              .onStart { showLoading() }
              .catchFailure { response -> showError(messageError =
  response.errorMessage.orEmpty()) }
              .collect { result ->
                  _uiState.update {
                      if (result.results.isEmpty()) SearchUIState.Empty
                      else SearchUIState.Success(result)
                  }
              }
      }
  }

  Also add debounce — prevents firing per-keystroke in the first place:

  init {
      viewModelScope.launch {
          _currentSearchText
              .debounce(300)
              .distinctUntilChanged()
              .collect { text ->
                  if (text.isBlank()) fetchMostPopularMovies()
                  else fetchMovieByQuery()
              }
      }
  }

  Then setCurrentSearchText just updates the flow — no manual fetchMovies() call
   needed from UI.

  Both fixes together: debounce cuts request count, job cancellation eliminates
  race condition for requests that do fire.
